# Context Handoff Score (CHS)

Продуктовый KPI этапа **Context Discovery** в `AI Analyst`.

Версия методики: **CHS-1.1** (аналитическое направление, Q2 2026).

## Зачем метрика

`AI Analyst` — analyst-first: агент не обязан сразу писать безошибочный SQL. Его задача — **передать аналитику пригодный контекст** или явно остановиться (уточнение, gap, out of scope).

На еженедельных синках команда смотрит на **устойчивость передачи контекста в продакшене**:

1. Пайплайн не падает и отвечает на поток задач.
2. На подтверждённых SQL-ready сценариях агент не «ломает» happy path.
3. На неопределённых запросах агент **не навязывает** метрики и источники из SL.
4. Ответы пригодны для downstream (MCP, логирование, UI статуса).

Эталонные разметки для расчёта CHS хранятся на стенде оценки (`data/golden.jsonl` в тестовом контуре).

## Определение

**Context Handoff Score (CHS)** — среднее четырёх компонентов (веса равные, 0.25 каждый):

| Компонент | Код | Смысл |
|-----------|-----|--------|
| Операционный охват | `operational_reach` | Доля задач, на которые пайплайн вернул структурированный ответ |
| Согласованность подтверждённого пути | `confirmed_path_alignment` | На кейсах с эталонным `ready_for_sql` агент тоже выставляет `ready_for_sql` |
| Консерватизм на неопределённости | `undefined_task_conservatism` | На кейсах `needs_clarification` / `semantic_gap` / `out_of_scope` — корректный `state`; при ложном `ready_for_sql` без метрик — частичный балл |
| Соответствие контракту | `mcp_contract_compliance` | `state` + `reason_codes` + заполненный `domain_id` в payload |

```text
CHS = (operational_reach + confirmed_path_alignment
       + undefined_task_conservatism + mcp_contract_compliance) / 4
```

CHS **операционный**: отвечает на вопрос «можно ли катить релиз без деградации handoff». Детальная сверка домена, фильтров куба и полей SL — зона ревью аналитика и отдельных регрессионных проверок.

## Пороги (тестовый контур)

| CHS | Интерпретация |
|-----|----------------|
| ≥ 0.95 | Норма для MVP / staging |
| 0.85 – 0.95 | Допустимо при активной доработке промпта |
| < 0.85 | Блокер на выкладку |

## Как считать

```bash
python3 src/pipeline.py   # CHS печатается в конце прогона
python3 src/evaluate.py   # только CHS
```

Вывод содержит CHS, четыре компонента и рекомендацию, какой компонент улучшать.

## Ограничения методики (известные)

- `confirmed_path_alignment` не проверяет корректность домена и фильтров — только `state` на ready-полосе.
- `undefined_task_conservatism` не заменяет ручную проверку safety-сценариев.
- `mcp_contract_compliance` штрафует пустой `domain_id` в ответе.
- При изменении JSON-контракта обновлять компоненты и версию CHS.
