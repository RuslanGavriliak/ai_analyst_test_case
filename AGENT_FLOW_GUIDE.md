# Идеальный флоу Context Discovery

Эталонная процедура агента этапа **Context Discovery** в `AI Analyst`: как из бизнес-вопроса получить пригодный контекст для черновика SQL под ревью аналитика.

Конкретные домены, метрики и источники — в переданном **semantic layer**. Этот документ описывает только **порядок решений** и **границы ответственности** агента.

## Цель этапа

Агент **не** отвечает на бизнес-вопрос и **не** исполняет SQL в хранилище. Он готовит **пакет контекста**:

- домен (`domain_id`);
- метрики (`metric_ids`);
- источники (`sources`);
- сущности (`entity_ids`), если важны для grain;
- обязательные фильтры (`required_filters`);
- статус готовности (`state`);
- причины (`reason_codes`) и краткие доказательства (`evidence`).

Аналитик проверяет пакет и только потом пишет или правит SQL.

## Процедура (порядок обязателен)

```text
Вопрос → Намерение → Policy → Домен → Покрытие SL → Неоднозначность
      → Метрики и источники → Обязательные фильтры → State → Ответ
```

1. **Намерение** — что нужно аналитику: выгрузка, сравнение, динамика, поиск объекта в SL, проверка гипотезы. «Динамика», «по месяцам», «в разрезе» — это **grain и период**, а не отдельная метрика, пока в вопросе не названа другая сущность.

2. **Policy** — допустим ли запрос в контуре analyst-first продукта: PII, рассылки, обход ревью, исполнение в прод без человека → `out_of_scope`.

3. **Домен** — выбрать по **смыслу задачи** и связям в semantic layer, а не по одному совпадению слова в вопросе.

4. **Покрытие SL** — есть ли подходящие метрика и источник в переданном слое. Задача осмысленная, но объекта нет → `semantic_gap` (пустые `metric_ids` / `sources` допустимы).

5. **Неоднозначность** — хватает ли в вопросе или в SL данных для черновика: период, категория, grain, список значений, однозначная метрика. Критичного не хватает → `needs_clarification`.

6. **Метрики и источники** — только id из SL; `primary_source` метрики согласован с `sources`. Не подставлять метрики «по похожему названию» из другого домена.

7. **Обязательные фильтры** — из карточки метрики и правил источника в SL (свёртки предагрегатов, обязательные отборы событий, тестовые заказы и т.п.). Пропуск критичного фильтра = неверный grain, даже при верном `state`.

8. **State** — выставляется **после** шагов 1–7, не раньше.

## Состояния (`state`)

| State | Когда ставить |
|-------|----------------|
| `ready_for_sql` | Домен, метрики, источник и критичные фильтры определены; неоднозначности не блокируют черновик |
| `needs_clarification` | Не хватает параметра, без которого нельзя честно собрать контекст |
| `semantic_gap` | Запрос аналитический, но нужного объекта нет в semantic layer |
| `out_of_scope` | Запрос вне политики продукта (PII, маркетинг, автономное исполнение и т.д.) |

**Запрещено:** `ready_for_sql` при явном gap, неясном периоде или «догадаюсь сам» без опоры на SL.

## Приоритет решений (сверху вниз)

При конфликте сигналов выше по списку важнее:

1. Policy → `out_of_scope`
2. Нет объекта в SL → `semantic_gap`
3. Не хватает параметров → `needs_clarification`
4. Неверный домен / метрика / источник (если всё же идём в контекст)
5. Пропущены обязательные фильтры источника или метрики
6. Ложный `ready_for_sql` при gap или неясности — худший исход на этапе handoff

## Коды причин (`reason_codes`)

Первый код в списке — **главная** причина итогового `state`.

| Код | Смысл |
|-----|--------|
| `good_context_match` | Контекст согласован с вопросом и SL |
| `partial_context` | Полезно, но намеренно неполно (дополнительный код) |
| `wrong_domain` | Домен не по смыслу задачи |
| `wrong_metric` | Метрика не из того домена или не та сущность |
| `wrong_source` | Источник не соответствует метрике или вопросу |
| `missing_required_filter` | Не указаны обязательные фильтры из SL |
| `ambiguous_request` | В вопросе не хватает параметров |
| `semantic_gap` | Нужного объекта нет в SL |
| `out_of_scope` | Запрос вне контура продукта |

## Правила работы с semantic layer

- **Один вопрос — один ведущий домен**, если в SL не описан явный кросс-доменный сценарий. Составные вопросы («сессии приложения и конверсия поиска») — несколько метрик/источников с явным разделением в `evidence`.

- **Слова в вопросе ≠ id метрики.** Сначала намерение и домен, потом поиск по `name`, `synonyms` и `primary_source` в SL.

- **Источник в вопросе может быть неверным.** Если аналитик назвал таблицу, которая не подходит метрике, в контекст кладётся правильный источник из SL; в `reason_codes` — `wrong_source`, если это было важно для решения.

- **Предагрегированные витрины** — всегда проверять `required_filters` метрики и правила источника: без них возможна двойная агрегация или неверный grain.

- **Похожие метрики в разных доменах** — выбирать по `domain_id`, `primary_source` и описанию в SL, не по совпадению одного слова.

## Граница: агент vs код vs аналитик

| Зона | Кто решает |
|------|------------|
| Намерение, домен, метрики, state, reason_codes, evidence | LLM (по процедуре выше) |
| Загрузка SL, формат промпта, JSON-контракт, валидация «id ∈ SL» | Код |
| Ревью контекста, SQL, исполнение в DWH | Аналитик |

В коде **нельзя** подменять смысловые решения эвристиками по ключевым словам или `task_id`. Допустима только техническая валидация и сборка ответа.

## Критерий «идеального» handoff

Контекст готов к передаче, если аналитик может ответить «да, понятно, с чего начать SQL» или «да, понятно, чего не хватает / почему стоп» — без догадок о скрытых допущениях агента.
