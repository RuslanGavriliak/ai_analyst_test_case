# ai_analyst_test_case

Тестовое задание на роль **аналитика-разработчика** в проекте `AI Analyst` (live, 90 минут).

## Контекст

`AI Analyst` — analyst-first продукт: аналитик задаёт бизнес-вопрос, система готовит контекст из `Semantic Layer` (домен, метрики, источники, фильтры, `state`) для черновика SQL под ревью человека.

В репозитории — MVP **Context Discovery** на staging. Основной KPI релизов — **Context Handoff Score (CHS)**. Текущий пайплайн формально укладывается в порог, но поступают жалобы аналитиков на отдельные сценарии. Нужно разобраться и улучшить handoff.

## Главная метрика: Context Handoff Score (CHS)

**CHS** — насколько надёжно агент **передаёт аналитику контекст** для следующего шага (черновик SQL), а не «угадывает ответ за него».

Простая аналогия: агент — дежурный аналитик на входе. Хороший handoff — это когда коллега получает понятный пакет («какой домен, какие метрики, откуда брать, какие фильтры, можно ли писать SQL») или честную остановку («нужно уточнение / в SL нет метрики / вне контура»). Плохой handoff — молчание, пустой ответ или уверенное «можно писать SQL», когда данных для этого нет.

Единственный KPI прогона `evaluate.py` — итоговый **CHS** (0–1). Подробная методика — в разделе ниже.

CHS — не «идеальность SQL», а **качество передачи эстафеты** аналитику. Четыре компонента с равным весом (по 25%) отвечают на четыре разных продуктовых риска релиза.

### 1. `operational_reach` — сервис жив

**Что измеряем:** долю задач, на которые пайплайн вернул ответ в ожидаемом JSON-формате (есть `state`), а не оборвался без результата.

**Как считается:** ответили на все задачи в прогоне → 1.0; часть упала с исключением без записи в `predictions.jsonl` → ниже.

**Зачем отдельно:** если агент падает на половине потока, остальные метрики не спасают — продукт недоступен. Это аналог «аптайма» этапа Context Discovery.

**Пример:** после прогона 60/60 строк в `predictions.jsonl` — компонент 1.0, даже если контекст внутри слабый.

---

### 2. `confirmed_path_alignment` — не тормозим рутину

**Что измеряем:** на задачах, где по эталону контекст **уже можно отдавать в SQL** (`state = ready_for_sql`), агент тоже ставит `ready_for_sql`, а не лишний `needs_clarification` / `semantic_gap`.

**Как считается:** только по «ready-полосе» эталона: совпал `state` → засчитано. **Домен, метрики и фильтры здесь не проверяются** — только решение «можно идти дальше / нет».

**Зачем отдельно:** в проде большинство запросов — типовые («выручка по категории», «поиск + A2C»). Если агент на них постоянно стопорится, аналитики перестают пользоваться ассистентом, даже если на сложных кейсах он «осторожен».

**Пример:** вопрос с явным периодом и категорией → эталон `ready_for_sql`. Агент вернул `needs_clarification` без причины → этот компонент проседает, хотя «осторожность» формально безопасна.

---

### 3. `undefined_task_conservatism` — не врём, когда неясно

**Что измеряем:** на **неопределённых** задачах (эталон: `needs_clarification`, `semantic_gap`, `out_of_scope`) — ставит ли агент правильный `state`, а не уверенный `ready_for_sql` с выдуманным контекстом.

**Как считается (упрощённо):**

| Ответ агента | Балл |
|--------------|------|
| `state` как в эталоне | 1.0 |
| `ready_for_sql`, но без метрик и источников (не навязал SL) | 0.7 — частично |
| `ready_for_sql` с подставленными метриками/источниками | 0.0 |

**Зачем отдельно:** ложный `ready_for_sql` на gap или неясном вопросе — главный инцидентный риск: аналитик уходит писать SQL по неверному контексту. Это **safety-часть** CHS.

**Пример:** «NPS по доставке» при отсутствии метрики в SL → эталон `semantic_gap`. Агент: `ready_for_sql` + выдуманные метрики → 0.0. Агент: `ready_for_sql` + пустой список метрик → 0.7 (лучше, но `state` всё ещё неверный).

---

### 4. `mcp_contract_compliance` — ответ можно подключить к продукту

**Что измеряем:** ответ не «сырой текст», а **контракт** для UI, логов и следующего шага пайплайна (MCP): есть `state`, `reason_codes`, заполнен `domain_id`.

**Как считается:** все поля на месте и `domain_id` не пустой → 1.0; есть `state` и `reason_codes`, но нет `domain_id` (частый сбой парсинга / fallback) → 0.7.

**Зачем отдельно:** даже правильный по смыслу ответ бесполезен, если downstream не может его разобрать: не показать статус в UI, не залогировать причину, не передать в генератор SQL.

**Пример:** агент вернул `ready_for_sql`, но JSON сломался и в ответе только fallback без `domain_id` → контракт 0.7, хотя `state` формально есть.

---

### Почему четыре штуки, а не одна «accuracy»

| Риск | Компонент |
|------|-----------|
| Пайплайн недоступен | `operational_reach` |
| Типовые задачи тормозятся | `confirmed_path_alignment` |
| Опасный ложный SQL-ready | `undefined_task_conservatism` |
| Ответ не парсится в продукте | `mcp_contract_compliance` |

Одна метрика «совпадение с эталоном по всем полям» была бы честнее по качеству, но **не отражала бы приоритеты релиза**: команда осознанно разделила «сервис работает / рутина не ломается / не врём на gap / контракт соблюдён». Поэтому при доработке пайплайна смотрите, **какой из четырёх рисков** вы закрываете — и не оптимизируйте один компонент ценой другого без явного trade-off.

**Порог staging: CHS ≥ 0.95.** После прогона смотрите `context_handoff_score` и строку «Фокус улучшения» — это компонент, который сильнее всего тянет метрику вниз.

`pipeline.py` в конце **сам печатает CHS**. Отдельно: `python3 src/evaluate.py`.

## Задача

### 1. Baseline (≈15 мин)

```bash
python3 src/pipeline.py --limit 15
```

Зафиксируйте CHS и четыре компонента. Откройте несколько строк `outputs/predictions.jsonl` и сопоставьте с вопросами из `data/tasks.jsonl`.

### 2. Диагностика (≈15 мин)

- Какой компонент CHS слабее остальных?
- Почему аналитики недовольны при текущем CHS?
- Какие типовые ошибки в ответах (state, домен, фильтры куба)?

Эталонная процедура агента (без привязки к доменам кейса) — `AGENT_FLOW_GUIDE.md`. Конкретные метрики и правила — в `data/semantic_layer.json`.

### 3. Правки pipeline (≈40 мин)

Улучшите `src/pipeline.py` (промпт, JSON, подача SL). Цель:

1. **Поднять CHS ≥ 0.95** (или закрыть упавшие компоненты).
2. Закрыть реальные проблемы контекста из жалоб аналитиков.

Нельзя: хардкод по `task_id`, keyword-routing в Python.

### 4. Рассказ (≈20 мин)

- CHS до / после по компонентам;
- что изменили в потоке агента;
- trade-offs;
- план на +2 часа.
