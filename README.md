# ai_analyst_test_case

Тестовое задание на роль **аналитика-разработчика** в проекте `AI Analyst` (live, 90 минут).

## Контекст

`AI Analyst` — analyst-first продукт: аналитик задаёт бизнес-вопрос, система готовит контекст из `Semantic Layer` (домен, метрики, источники, фильтры, `state`) для черновика SQL под ревью человека.

В репозитории — MVP **Context Discovery** на staging. Основной KPI релизов — **Context Handoff Score (CHS)**. Текущий пайплайн формально укладывается в порог, но поступают жалобы аналитиков на отдельные сценарии. Нужно разобраться и улучшить handoff.

## Главная метрика: Context Handoff Score (CHS)

**CHS** — насколько надёжно агент **передаёт аналитику контекст** для следующего шага (черновик SQL), а не «угадывает ответ за него».

Простая аналогия: агент — дежурный аналитик на входе. Хороший handoff — это когда коллега получает понятный пакет («какой домен, какие метрики, откуда брать, какие фильтры, можно ли писать SQL») или честную остановку («нужно уточнение / в SL нет метрики / вне контура»). Плохой handoff — молчание, пустой ответ или уверенное «можно писать SQL», когда данных для этого нет.

Единственный KPI прогона `evaluate.py` — итоговый **CHS** (0–1). Подробная методика — в `CONTEXT_HANDOFF_SCORE.md`.

| Компонент | Вес | Что измеряем по-продуктовому |
|-----------|-----|------------------------------|
| `operational_reach` | 25% | **Сервис отвечает.** На сколько вопросов агент вообще вернул структурированный ответ, а не «упал» по пути. |
| `confirmed_path_alignment` | 25% | **Не ломаем рутину.** На понятных задачах, где контекст уже можно собирать, агент не блокирует аналитика лишними статусами и доходит до `ready_for_sql`. |
| `undefined_task_conservatism` | 25% | **Не врём в неопределённости.** Если вопрос неясен, метрики нет в SL или запрос вне контура — агент не выдаёт уверенный SQL-ready контекст «из головы». |
| `mcp_contract_compliance` | 25% | **Ответ пригоден для продукта.** В ответе есть статус (`state`), причины решения (`reason_codes`) и минимально заполненный контекст (в т.ч. `domain_id`) — чтобы UI, логи и следующий шаг пайплайна могли это обработать. |

**Порог staging: CHS ≥ 0.95.** После прогона смотрите `context_handoff_score` и строку «Фокус улучшения» — это компонент, который сильнее всего тянет метрику вниз.

`pipeline.py` в конце **сам печатает CHS**. Отдельно: `python3 src/evaluate.py`.

## Задача

### 1. Baseline (≈15 мин)

```bash
python3 src/pipeline.py --limit 15
```

Зафиксируйте CHS и четыре компонента. Откройте несколько строк `outputs/predictions.jsonl` и сопоставьте с вопросами из `data/tasks.jsonl`.

### 2. Диагностика (≈15 мин)

- Какой компонент CHS слабее остальных?
- Почему аналитики недовольны при текущем CHS?
- Какие типовые ошибки в ответах (state, домен, фильтры куба)?

Ориентир по процедуре агента — `AGENT_FLOW_GUIDE.md`.

### 3. Правки pipeline (≈40 мин)

Улучшите `src/pipeline.py` (промпт, JSON, подача SL). Цель:

1. **Поднять CHS ≥ 0.95** (или закрыть упавшие компоненты).
2. Закрыть реальные проблемы контекста из жалоб аналитиков.

Нельзя: хардкод по `task_id`, keyword-routing в Python.

### 4. Рассказ (≈20 мин)

- CHS до / после по компонентам;
- что изменили в потоке агента;
- trade-offs;
- план на +2 часа.
